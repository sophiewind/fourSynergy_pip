fourSig Tutorial

fourSig is a suite of programs used to identify significant chromosomal interactions from 4C-seq data. Currently it works with data in either SAM, BAM or Bowtie formats.

This tutorial will teach you how to use the fourSig suite of programs. In it we will analyze the interactions that take place at the Beta-Globin locus in mice.

  1. Download and unpack fourSig

    If you haven't already done this, download the latest version of fourSig from the Sourceforge website: https://sourceforge.net/projects/foursig/

    Now unpack it 

    
shell$ tar -xzvf fourSig.tar.gz

  2. Get the data

    There are two ways to get the data: 1) the easy way and 2) the hard way.

    • The easy way.

      Use the pre-formatted data that comes with this tutorial. If you choose this option go straight to the analysis section.

    • The hard way.

      Use the raw data. The plus side of using raw data is that it will be more like what you will have to do with your own data. The downside to using the raw data is that it is in BAM format and you'll need samtools to read it. However, there's a good chance you already have it installed.

      If you choose to download the raw data, you can read about it here:

      http://r3cseq.genereg.net/Site/index.html

      And the files are here:

      http://r3cseq.genereg.net/download/Beta-major-prom-FL.bam

      http://r3cseq.genereg.net/download/Beta-major-prom-FB.bam

  3. Format the data for analysis

    If you downloaded the BAM files, the next thing to do is to create TAB formatted files. TAB format is just a tab delimited file that contains an index of all the possible genomic fragments that the reads could map to. It also keeps track of how many reads map to them, and also whether or not the fragment is mappable (see our paper for details on how we determine if a fragment is mappable - in short, it depends on the length of the fragment, as well as how it will be digested by the 3C and 4C restriction enzymes). 

    
shell$ ./bamToReTab.pl -M -o "Y" 18 700 20 Beta-major-prom-FB.sam mm9_re_sites/b6_hind3_sites.txt mm9_re_sites/b6_nla3_sites.txt NONE > Beta-major-prom-FB.tab

shell$ ./bamToReTab.pl -M -o "Y" 18 700 20 Beta-major-prom-FL.sam mm9_re_sites/b6_hind3_sites.txt mm9_re_sites/b6_nla3_sites.txt NONE > Beta-major-prom-FL.tab

    As you see, bamToReTab.pl takes a bunch of arguments. Here is what they are (NOTE, you can read about additional options by simply typing ./bamToReTab.pl at the command line without any other options are arguments):

    • -M : This specifies to use the mouse mm9 genome. You can specify the human hg19 genome by using -H instead. If you want to use another genome, there are just a few easy edits to the code you need to do. Just open up bamToReTab.pl and follow the directions.
    • -o "Y" : This specifies that we want to omit analysis of chromosome Y. You can omit other chromosomes simply by adding them to the list, i.e. -o "1 2 3" would omit chromosomes 1, 2 and 3 from the analysis.
    • Argument 1: The length of the sequenced read that was in the anchor fragment. This includes the length of the 4C primer that you used as well as any additional basepairs that were sequenced prior to the 3C or 4C cut sites. In this case, the value 18 was inferred from the fact that the reads were 76 bp long, but only 58 bp were aligned.
    • Argument 2: The maximum number of bp submitted for sequencing after the size selection step in 4C.
    • Argument 3: The minimum number of bp submitted for sequencing.
    • Argument 4: The BAM file.
    • Argument 5: A file with locations of the 3C restriction enzyme. Note, before generating one of these files on your own, make sure there isn't one already in the mm9_re_sites directory. These were created using the mm9 mouse sequence. If you want to create files for hg19 or another genome, you can use motifFinder.pl, which is provided as part of the fourSig suite of programs. Type motifFinder.pl -h to learn how to use it.
    • Argument 6: A file with locations of the 4C restriction enzyme.
    • Argument 7: A file with locations of the linearization restriction enzyme, or NONE, if there was no linearization step in the 4C protocol.

    bamToReTab.pl pumps its output to standard out (STDOUT), so make sure you direct it to a file (a seen in the examples above with the > character).

    If you data is in Bowtie format, just use bowtieToReTab.pl. It runs the same way.

  4. Analysis

    Start up R. (Download R if you don't already have it installed.) 

    
shell$ R

    Or, if you prefer RStudio, you can use that.

    Now load in the fourSig analysis suite: 

    
> source("fourSig.R")

    NOTE: In the examples below we are only using 100 iterations so that you can relatively quickly get an idea of how the program works. For your own research, consider increasing the number of iterations to 1,000.

    First, let's set the window size to 1, so that we are analyzing the individual fragments. 

    
> fl.sig.w1 <- fourSig(filename="Beta-major-prom-FL.tab", chr=7, cis.only=TRUE, window.size=1, iterations=100, fdr=0.01, fdr.prob=0.05)

    fourSig takes a handful of arguments. Here is an explanation of the ones you see here, plus some more....

    filenameThe name of a TAB file you created earlier.
    chrThe name of the cromosome that the viewpoint is on.
    cis.only[TRUE | FALSE] (Default =FALSE) If TRUE only generate stats for CIS interactions.
    window.sizeThe number of 3C fragments allowed in the windows.
    iterationsThe number of random shuffles used to generate a distribution of signifcance cut-offs required to achieve a given FDR.
    fdrThe FDR used define the cut-offs.
    fdr.probThe percentage of iterations that had a cut-off larger than the final threshold of significance.

    ... and these are extra options....

    trans.only[TRUE | FALSE] (Default = FALSE) If TRUE only generate stats for TRANS interactions.
    mask.start (Default = NULL) Defines the start point for a region you want to mask out. NOTE: We are not masking out the veiw point because this is the region that we are primarily interested in. However, you may find that your own analysis requires you to mask out the view point. If your intial analysis provides unsatisfying results, mask out the view point and re-run the analysis.
    mask.end (Default = NULL) Defines the end point for a region you want to mask out.
    invert.mask[TRUE | FALSE] (Default = FALSE). When set to FALSE, the region between mask.start and mask.end is masked out. When set to TRUE the regions outside of the area defined by mask.start and mask.end are masked out.
    out.fileUse this to specify a filename you want the output to be written to.
    only.mappable[TRUE | FALSE ] (Default = TRUE). This option determines whether or not to use only the mappable fragments in the analysis, or all of the fragments, regardless of mappability.

    fourSig returns the interacting domains in a dataframe with columns for chromosome number (chr), domain start (start), domain end (end) and significance category (category). Here are the first 6 rows of results (see below for an explanation for what the significance categories correspond to): 

    
> head(fl.sig.w1)
  chr   start     end category
1   7 3146334 3146334        3
2   7 3610775 3610775        3
3   7 4456451 4456451        3
4   7 4566788 4566788        3
5   7 4670113 4670113        3
6   7 4689088 4689088        3

    The significance categories explained: Although all categories (1 through 3) represent statistically significant windows, we rank them based on additional criteria to give users a sense of what’s going on. Are there multiple fragments in the window with lots of reads? Or is only one fragment within the window doing all the work to make it significant?

    • Category 1 - This means that there were multiple fragments within the window that had a statistically significant number of reads mapping to them. In practical terms, if you are going to validate with FISH or 3C, start with the “category 1” regions. With a window of 100, this may mean that there are multiple binding sites within the window, or it could mean that there is a single binding site that spills over in to neighboring fragments because of incomplete digestions. You can tell by looking at the “tab” file that you created.

    • Category 2 - If you replace the fragment with the most reads in the window with the average of the remaining fragments, the window is still significant.

    • Category 3 - This means that the only way for a window to contain a statistically significant number of reads is to include all of the reads on the fragment with the most of them. If you remove the one fragment with the most reads, the window becomes insignificant, and if replace that fragment with the average, it’s still insignificant. NOTE - based solely on intuition, category 3 windows are the ones we have the least confidence in and we feel might require the largest sample size to validate with FISH or 3C. However, it could be that you have a very strong interaction that is just very narrow and there were not many incomplete digestions. In this case - look at the read counts within that window (in the tab file). If you see a single one with tons of reads, that might be a good hit. On the other hand, it might also be a repetitive region - so be sure to check before you commit a lot of resources to validation!

  5. Graphing the results in R

    First, figure out how many reads are in each window. This step allows you to display the data the way fourSig saw it, but it can also let you visually inspect other window resolutions before commiting to a full analysis. 

    
> fl.w1 = readsToWindows(file="Beta-major-prom-FL.tab", chr=7, window.size=1, only.mappable=FALSE, mask.start=110975750, mask.end=110988565)

    NOTE: Here we set only.mappable to FALSE because we want to draw all of the fragments across the chromosome.

    NOTE: The example here is for CIS interactions . If you want to graph TRANS interactions then you need to set chr to "all" (include the quotes) and exclude mask.start and mask.end or set them to NULL.

    Now graph it! 

    
> plotReads(fl.w1, fl.sig.w1, x.start=110940000, x.end=111040000, connect.points=TRUE, y.max=8500, mask.start=110975750, mask.end=110988565, draw.reads.first=TRUE)

    And that command also has a lot of arguments....

    The first two should be self explanatory (the read counts per window, and the results from the fourSig analysis).

    use.log [TRUE | FALSE] (Default = FALSE) Scale the y-axis with log10.
    x.start, x.end Define the region we want to look at. Leave these out if you want to look at the entire chromosome.
    connect.points [TRUE | FALSE] (Default = FALSE) This effects how the window counts will be displayed.
    y.max This is the maximum value you want on the y-axis. It's useful if you want to draw multiple graphs that all have the same y-axis. Leave it off otherwise.
    draw.reads.first [TRUE | FALSE] (Default = TRUE) This tells the program to draw the window counts before it draws the significance regions. Otherwise it draws the significance regions first, and the read counts on top. Change this according to how things look to you. Sometimes things look better one way, sometimes they look better the other.
    show.fragments [TRUE | FALSE] (Default = TRUE) This displays a map of the 3C fragments under the graph. This is useful for graphs that are zoomed into a particular region. However, you may want to turn this feature off when looking at an entire chromosome.
    mask.start (Default = NULL) Defines the start point for a region you want to mask out. NOTE: Here we are masking out the view point fragment because we are not interested what this fragment looks like.
    mask.end (Default = NULL) Defines the end point for a region you want to mask out.

    Now save the results as a PDF: 

    
> pdf(file="fl_results.pdf", width=8, height=4)
> plotReads(fl.w1, fl.sig.w1, x.start=110940000, x.end=111040000, connect.points=TRUE, y.max=8500, mask.start=110975750, mask.end=110988565, draw.reads.first=TRUE)
> dev.off()

  6. Graphing the results in the UCSC Genome Browser

    Just like graphing the results in R, we first need to figure out how many reads are in each window. If you already did this step, you can skip it, otherwise, here's the command (see the "Graphing the Results in R" section for more details in how this command works): 

    
> fl.w1 = readsToWindows(file="Beta-major-prom-FL.tab", chr=7, window.size=1, only.mappable=FALSE, mask.start=110975750, mask.end=110988565)

    NOTE: The example here is for CIS interactions . If you want to graph TRANS interactions then you need to set chr to "all" (include the quotes) and exclude mask.start and mask.end or set them to NULL. Now save the windowed reads to a bedGraph file, which you can upload to the UCSC Genome Browser: 

    
> saveWindowsToBedGraph(window.data=fl.w1, filename="fl_reads.bedGraph", trackname="fl_reads")
    The arguments for saveWindowsToBedGraph() are defined as follows:

    window.dataSet this to the output of readsToWindows().
    filenameThe name of the new bedGraph file that you want to create.
    trackname(optional)The track name that will be displayed in the UCSC genome browser.

    Now we need to save the output from fourSig() as a BED file that will show were the reads were significantly enriched. 

    
> saveSigAsBed(sig.data=fl.sig.w1, window.data=fl.w1, filename="sig_interactions.bed", trackname="sig_interactions", mask.start=110975750, mask.end=110988565)
    The arguments for saveSigAsBed() are as follows:

    sig.dataSet this to the output of fourSig().
    window.dataSet this to the output of readsToWindows().
    filenameThe name of the new bed file that you want to create.
    trackname(optional)The track name that will be displayed in the UCSC genome browser.
    mask.startThis is the same as for the other functions.
    mask.endThis is the same as for the other functions.

    Once you have created the bedGraph and BED files, you can upload them to the UCSC genome browser.
